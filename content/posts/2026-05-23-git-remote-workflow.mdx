## 들어가며

Git을 혼자 쓸 때는 로컬 저장소만으로 충분하지만, 팀과 협업하는 순간 **리모트 저장소**라는 새로운 차원이 생긴다. 많은 개발자가 `git pull`과 `git push`를 자동으로 누르지만, 그 속에서 일어나는 일이 무엇인지—특히 `git fetch`의 역할이 무엇인지—명확히 이해하지 못하면 충돌 해결, 히스토리 이해, 브랜치 병합이 헷갈린다.

이번 편에서는 **세 개의 트리**가 아닌 **세 개의 저장소**(`local` ↔ `remote` ↔ `origin`)의 관계와, fetch·pull·push가 정확히 어떻게 작동하는지 파헤친다.

## 1. 저장소 3개의 구조: 로컬 ↔ 리모트 트래킹 ↔ 원격 저장소

Git에서 다루는 저장소는 사실 3개다.

```
[Local Working Tree]
        ↓ (git add)
[Local Staging Area]
        ↓ (git commit)
[Local Repository] ← HEAD, main (로컬 브랜치)
        ↕ (fetch / push)
[Remote Tracking Branch] ← origin/main (추적 브랜치, 로컬에만 존재)
        ↕ (fetch / push)
[Remote Repository] ← origin의 실제 main (GitHub 등)
```

핵심: **`origin/main`은 로컬에 있다.** GitHub의 `main`이 아니라, GitHub의 상태를 *로컬에 복제한 스냅샷*일 뿐이다.

`git branch -a`를 치면:
```
* main (로컬 브랜치)
  remotes/origin/main (리모트 트래킹 브랜치)
```

## 2. fetch: "리모트의 새로운 커밋을 받아온다"

`git fetch origin`은:
1. GitHub의 `origin/main`과 로컬의 `origin/main`을 비교
2. 새로운 커밋들을 로컬로 다운로드
3. 로컬의 `origin/main` 포인터만 업데이트
4. **로컬 `main` 브랜치는 건드리지 않음**

```bash
$ git fetch origin
```

결과:
- `origin/main` → 최신 상태로 업데이트
- `main` → 그대로 (변화 없음)
- Working Tree → 그대로

즉, fetch 후에도 코드는 변하지 않는다. 단지 "리모트에서 뭐가 새로 올라왔는지"만 알 수 있을 뿐이다.

```bash
$ git log origin/main..main  # 로컬이 얼마나 앞서 있나?
$ git log main..origin/main  # 리모트가 얼마나 앞서 있나?
```

## 3. pull: fetch + merge (또는 rebase)

`git pull origin main`은 사실:
```bash
git fetch origin
git merge origin/main  # 기본값
# 또는
git rebase origin/main  # git config pull.rebase true 설정 시
```

즉, fetch 후 자동으로 merge(또는 rebase)를 실행한다.

```bash
$ git pull origin main
# ↓ 이것과 같다
$ git fetch origin && git merge origin/main
```

**merge vs rebase 선택:**
- `git pull --merge` (기본): 3-way merge로 병합 커밋 생성
- `git pull --rebase`: rebase로 선형 히스토리 유지

설정으로 기본값을 정할 수 있다:
```bash
git config pull.rebase true  # 이후 모든 pull은 rebase
```

## 4. push: 로컬 커밋을 리모트로 보낸다

`git push origin main`은:
1. 로컬 `main`의 새로운 커밋들을 찾는다
2. 로컬 `origin/main`과 비교해 새로운 것들을 GitHub로 전송
3. GitHub의 `main`과 로컬의 `origin/main` 모두 업데이트

```bash
$ git push origin main
```

**fast-forward가 불가능하면 push가 거절된다:**
- 리모트가 더 이상 로컬의 ancestor가 아니면 충돌
- 해결책: `git pull` → 로컬 업데이트 → `git push`

```bash
$ git push origin main
# error: failed to push some refs to '...'
# hint: Updates were rejected because the tip of your current branch is behind
$ git pull origin main
$ git push origin main  # 이제 가능
```

## 5. 실전 협업 워크플로

### 패턴 1: fetch → 확인 → merge

```bash
# 1. 다른 팀원의 작업 확인
git fetch origin
git log origin/main --oneline -5

# 2. 로컬 main으로 돌아가서
git checkout main

# 3. 안전하게 병합 (merge)
git merge origin/main
```

### 패턴 2: pull → 작업 → push

```bash
# 1. 최신 상태 동기화
git pull origin main

# 2. 새로운 브랜치에서 작업
git checkout -b feature/my-work

# 3. 커밋
git add .
git commit -m "feat: 새 기능"

# 4. 리모트로 보내기
git push origin feature/my-work

# 5. GitHub에서 PR 생성 → 리뷰 → merge
```

### 패턴 3: 실수했을 때 되돌리기

```bash
# A. 이미 로컬에서 commit까지 했지만 아직 push 안 함
git reset --soft HEAD~1  # 커밋 취소, 변경 사항은 staging 상태로

# B. 이미 push해버렸다면?
git revert HEAD  # 새 커밋으로 되돌리기 (히스토리 유지)
git push origin main
```

## 6. 리모트 트래킹 브랜치 이해하기

`git branch -r`로 리모트 트래킹 브랜치 목록을 본다:
```bash
$ git branch -r
  origin/main
  origin/develop
  origin/feature/x
```

이들은:
- **로컬 저장소 안에만 존재** (원격 저장소에서 복제된 스냅샷)
- `git fetch`할 때마다 최신으로 동기화
- 건드리면 안 됨 (fetch로만 업데이트 받아야 함)

```bash
# 리모트 브랜치에서 로컬 브랜치 만들기
git checkout -b local-feature origin/feature/x
# 또는 (최신 Git)
git checkout feature/x  # 자동으로 origin/feature/x 추적
```

## 7. 충돌 해결

### 병합 중 충돌

```bash
$ git pull origin main
# CONFLICT (content merge): Merge conflict in file.js

# 1. 파일 편집해서 충돌 해결
# 2. 파일 재검토 후
git add file.js
git commit -m "Merge branch 'main' into local-main"
git push origin main
```

### fast-forward 실패

```bash
$ git push origin main
# error: failed to push

# 원인: 리모트가 로컬보다 최신 상태
$ git pull origin main  # merge 또는 rebase
$ git push origin main  # 재시도
```

## 마무리

Git의 협업은 **fetch·pull·push 3가지 명령으로 정리된다:**
- **fetch**: 리모트에서 정보만 가져온다 (코드는 안 바뀜)
- **pull**: fetch + merge(또는 rebase), 즉 코드까지 동기화
- **push**: 로컬 커밋을 리모트로 보낸다

로컬 저장소의 3개 영역(working tree, staging, repository) 외에, **리모트 트래킹 브랜치라는 4번째 참조점**이 있다는 것을 이해하면, 충돌도 두렵지 않고 협업이 직관적이 된다.
